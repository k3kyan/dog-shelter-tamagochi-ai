ChromaDB: vector database (!!!!!) that stores and searches embeddings
    embeddings - a numerical representation (like vectors) of data (like text, images, or audio) that captures its semantic meaning
sentence-transformers: converts text to vectors (embeddings)
    torch - sentence-transformers requires PyTorch (wont be using pytorch in this project directly)
langchain: privdes the RecursiveCharacterTextSplitter for chunking
    chunking - breaking down large documents into smaller, manageable text segments (chunks) during indexing. improves retriever efficiency by allowing semantic search to fetch precise, context-aware info instead of the entire document. improves rag system accuracy.
    using Recursive Character Text Splitting instead of Contextual Retrieval or Fixed-size (can cut off sentences (bad)) chunking 
        actually, i think im gonna add Contextual Retrieval 
langchain-groq: calls Groq to generate context sentences per chunk
python-dotenv: loads GROQ_API_KEY from .env

1. get the api key for Groq
console.groq.com

2. Chunk article text
3. Contextual Retrieval (add LLM-generated context to each chunk using Groq)
4. embed with sentence-transformers
5. store in ChromaDB

BIG FIX: The bottleneck was generate_content() bc sending the FULL ARTICLE to groq, it can mess up Groq's tokens-per-minute rate limit, which causes it to throttle and makes each call take 20-30 seconds. So i truncated it bc i only need enough surrounding context for the chunk, not the entire article.
